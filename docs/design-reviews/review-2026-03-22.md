## 1. First Impression (Gut Reaction)

**What the site communicates at a glance:** A serious, dark-mode ops dashboard built by someone who values density and information hierarchy.

**What the eye is drawn to first:**
1. The amber "needs attention" banner — smart, it's the highest-signal element
2. The metric cards row (Projects / Active / Budget / Agents)
3. The Project Progress table — dense but scannable

**One-word verdict:** Competent

**Gut reaction:** This doesn't look AI-generated — it looks like an internal tool built by an engineer who's spent real time in Linear and Vercel's dashboard. The `--radius: 0` decision is the strongest single design choice: it immediately separates this from the rounded-everything aesthetic of AI-generated UIs. The information density is high but deliberate. The biggest risk is that it's so restrained it verges on austere — there's no moment of delight or surprise anywhere. It's all business, all the time.

## 2. Inferred Design System

### Fonts
- **Body:** `-apple-system, BlinkMacSystemFont, "SF Pro Text", "SF Pro Display", system-ui, sans-serif` — Apple system stack. Good choice for a Mac-first internal tool.
- **Mono:** `"SF Mono", ui-monospace, SFMono-Regular, Menlo, Monaco, Consolas` — Used consistently for numbers, IDs, budget figures.
- **Font families:** 2 (system sans + system mono). Clean.
- **Heading scale:** H1 at `text-sm font-semibold uppercase tracking-wider` (~14px), large numbers at `text-2xl sm:text-3xl` (~24-30px). Section headers at `text-[11px] uppercase tracking-[0.16em]`. The headings are all small-caps labels — there's no traditional heading hierarchy. This is intentional (ops dashboard, not marketing page) but means the page has no typographic drama.

### Colors
- **Background:** `oklch(0.145 0 0)` — near-black, zero chroma. Pure dark mode.
- **Card/secondary:** `oklch(0.205-0.269 0 0)` — desaturated grays for layering.
- **Accent colors:** Semantic only — green (active/done), amber (warning/pending), red (error/destructive), cyan (running/in-progress), blue (completed), violet (proposed/review). No brand color.
- **Unique non-gray colors:** ~8 semantic colors. Well within bounds.
- **Palette assessment:** Coherent. Every color means something. No decorative color usage.

### Spacing
- **Base unit:** Tailwind's 4px grid (`gap-1` = 4px, `gap-2` = 8px, `p-4` = 16px, `p-6` = 24px).
- **Common values:** 4, 8, 12, 16, 24 — clean 4px rhythm.
- **Metric card gap:** `gap-1 sm:gap-2` (4-8px). Tight. Intentional density.
- **Section spacing:** `space-y-6` (24px) between major sections. Consistent.

### Border Radius
- **`--radius: 0`** globally. Sharp corners on everything — cards, buttons, inputs, progress bars.
- **Exceptions:** `rounded-full` on badges and status dots (semantic — they're pills/circles). `rounded-md` on buttons (via the Button component — **inconsistency noted**, see findings).
- **Assessment:** The zero-radius decision is the single strongest design opinion in this dashboard. It's bold and correct for an ops tool.

### Layout
- **Grid-based:** Yes — `grid-cols-2 xl:grid-cols-4` for metrics, responsive table for projects.
- **Sidebar:** Fixed 240px (`w-60`), full-height, border-right separated.
- **Content area:** `p-4 md:p-6`, scrollable, flex-1.
- **Assessment:** Standard sidebar + content layout. Well-executed, nothing surprising.

## 4. Top 5 Design Improvements

### 1. Mobile navigation is missing
**What:** The 240px sidebar has no mobile toggle/drawer behavior. On phones, the sidebar would either overflow or force horizontal scroll.
**Why it matters:** The PWA meta tags (`apple-mobile-web-app-capable`, `viewport-fit=cover`) suggest mobile use is intended. Without a collapsible sidebar, the dashboard is desktop-only in practice.
**Severity:** High
**Category:** Responsive

### 2. Button border-radius inconsistency
**What:** Global `--radius: 0` sets sharp corners everywhere, but the Button component uses `rounded-md` (6px radius). StatusBadge uses `rounded-full`. The OrgChart container uses `rounded-lg`. These break the zero-radius system.
**Why it matters:** The `--radius: 0` is the dashboard's strongest design choice. Every exception weakens it. Buttons should be sharp-cornered to match. Badges/pills can stay `rounded-full` (they're semantically different — they're pills, not containers).
**Severity:** Medium
**Category:** Spacing / Consistency

### 3. No focus-visible states for keyboard navigation
**What:** Interactive elements (`SidebarNavItem`, `TabsTrigger`, table rows, buttons) have hover states but no `focus-visible` outlines.
**Why it matters:** Keyboard navigation is invisible. For an ops tool where a user might tab through approvals quickly, this is a real usability gap.
**Severity:** Medium
**Category:** Interaction States

### 4. Hardcoded "Agents: 9" on overview
**What:** The fourth metric card on the Overview page hardcodes `value="9"` instead of deriving from data.
**Why it matters:** When you add or remove an agent, the dashboard lies to you. In an ops tool, stale data erodes trust in everything else the dashboard shows.
**Severity:** High
**Category:** Content Quality

### 5. Section headers lack differentiation
**What:** Every section uses the same `text-[11px] uppercase tracking-[0.16em] text-muted-foreground` treatment — "Project Progress", "Recent Issues", "Per-Agent Breakdown", "Budget Policy", etc. There's no hierarchy between a page title and a section label.
**Why it matters:** When everything is the same weight, nothing guides the eye. The page title ("Dashboard") is `text-sm font-semibold uppercase tracking-wider` — barely distinguishable from section headers. On dense pages (ProjectDetail with 7 tabs), the visual flatness makes scanning slower.
**Severity:** Medium
**Category:** Hierarchy / Typography

## 5. Additional Findings (Polish)

- **Sidebar version string** (`OpenClaw v2026.3.13`) is hardcoded — should pull from package.json or build-time env var.
- **Search button** in sidebar has no implementation — it renders but does nothing. Either implement or remove.
- **`prompt()` for rejection comments** (in `handleReject`) — native browser prompt breaks the visual language. Should be an inline input or modal.
- **Empty state icon containers** use `bg-muted/50 p-4` with no border-radius, creating a visible square. This is correct per the `--radius: 0` system but the padding is generous — the icon looks small inside a large gray box.
- **Table header text** uses `text-[10px]` — this is very small (10px). At this size, readability depends heavily on the display. Consider `text-[11px]` minimum.

## 6. Recommendations

1. **You don't have a design system defined.** Run `design-create` to build one — then `design-qa` can fix against it. The good news: this dashboard already has strong implicit opinions (zero radius, semantic color, density-first). A DESIGN.md would codify them and prevent drift.

2. **To fix the mechanical issues** (border-radius inconsistency, missing focus states, hardcoded values): Run `design-qa` to auto-fix.

3. **Mobile navigation** is the single highest-impact improvement if mobile use is intended. If it's desktop-only, remove the PWA meta tags to set expectations correctly.

## 7. Paperclip Comparison — Styling Gaps

Reference: [Paperclip](https://github.com/paperclipai/paperclip) agent detail page screenshot (CEO agent view).

Same product category, same dark-mode aesthetic, same sidebar+content layout. These gaps are purely about visual/styling treatment — not missing features.

### Gap 1: Agent name is too small — no page-level type hierarchy
**Paperclip:** The agent name "CEO" is rendered at ~28-32px, bold, immediately commanding the page. It's the largest text element on screen. Clear what page you're on from across the room.
**Us:** Agent name is `text-xl` (20px). Page titles on overview/agents/costs are `text-sm uppercase` (14px). Everything is roughly the same size. There's no moment where the typography says "THIS is the important thing on this page."
**Fix:** Page entity names (agent name, project name) should be `text-2xl` or `text-3xl`. The small-caps label style works for section headers, not for the primary entity on a detail page.

### Gap 2: Agent avatar/icon is undersized and flat
**Paperclip:** The agent icon is a ~48px rounded square with a subtle dark fill and a briefcase icon inside — it has visual weight, reads as an identity marker. Sits proportionally next to the large name.
**Us:** `h-12 w-12 bg-accent` with a plain emoji or first-letter initial. Same 48px size, but the `bg-accent` (which resolves to `oklch(0.269 0 0)`) is barely distinguishable from the page background. The avatar disappears instead of anchoring the header.
**Fix:** Give the avatar more contrast — either a slightly brighter fill, a border, or a colored tint that ties to the agent's status. The avatar should be the visual anchor of the header row.

### Gap 3: Header action buttons row is missing visual weight
**Paperclip:** The header has a row of outlined/ghost buttons ("Assign Task", "Invoke", "Pause") plus an "idle" status pill — all right-aligned, creating a clear action zone. The buttons use a subtle border treatment that separates them from the background without being heavy.
**Us:** The AgentHeader has no right-side content at all. Even ignoring the functionality, the header layout is left-heavy — name/role/projects all stack on the left, the right side is empty space. The header feels unbalanced.
**Fix:** Even without wiring up actions, the header needs a right-aligned zone — at minimum the status badge and metadata (last seen, run count) should float right to balance the composition.

### Gap 4: Sidebar lacks depth layering between sections
**Paperclip:** The sidebar has visible separation between section groups — PROJECTS, WORK, AGENTS, COMPANY each have distinct spacing and the section labels create clear visual breaks. Individual agents appear with inline status badges ("1 live" in green), giving the sidebar texture and information density.
**Us:** The sidebar sections (top nav, Projects, Work) use `gap-4` (16px) between groups, which is fine. But the content within sections is visually flat — every item is the same height, same weight, same treatment. Projects get colored dots (good) but nav items and project items look nearly identical otherwise. The sidebar reads as a uniform list rather than a structured hierarchy.
**Fix:** Increase contrast between section types. Consider giving nav items slightly different padding or weight than project items. The section labels (`text-[10px] uppercase tracking-widest font-mono text-muted-foreground/60`) are almost invisible — they could be slightly more prominent.

### Gap 5: "Latest Run" card has no visual distinction from other content
**Paperclip:** The "Latest Run" section uses a subtle card treatment with a slightly elevated background, a "View details" link right-aligned in the header, and inline metadata (status badge, duration, trigger type, time ago) all on one scannable line. It reads as a distinct, self-contained status card.
**Us:** All content sections use the same `border border-border p-4` treatment. The "Current Work" section, "Waiting On" section, "Projects" section, "Identity" section — they're all identical bordered boxes. Nothing visually signals which section is more important than another.
**Fix:** The most important status section (whatever surfaces the agent's current state) should have a differentiated card treatment — either a slightly elevated background (`bg-card` or `bg-accent/20`), a left-border accent color, or a larger internal heading. Not everything can be the same box.

### Gap 6: Breadcrumb separator style
**Paperclip:** Uses `>` as the breadcrumb separator ("Agents > CEO"), which reads as directional navigation.
**Us:** Uses `/` as the separator ("Agents / CEO"), which reads more like a file path. Minor, but the `>` convention is more standard in app navigation and visually suggests hierarchy/drilling down.
**Fix:** Replace `/` separator with `>` or a chevron icon in breadcrumbs.

### Gap 7: Issue row density and alignment
**Paperclip:** Issue rows in "Recent Issues" have tight, scannable density — issue ID left-aligned in mono (`PAP-61`), title taking remaining space, status badge right-aligned (`done` in green). The rows are compact, ~36px height, with clear left-to-right reading flow.
**Us:** Our `EntityRow` is similar in structure but adds more visual noise — leading slot (priority dot + status circle), identifier, title, trailing slot (status badge + assignee + time ago + arrow icon). The trailing slot has 4 elements, which crowds the right side. The arrow icon (`ArrowRight`) on every row is visual noise — it doesn't add information.
**Fix:** Reduce trailing elements. Drop the `ArrowRight` icon — the `cursor-pointer` and `hover:bg-accent/50` already signal clickability. Consider hiding the assignee on narrow viewports.

### Summary: Styling Gaps

| Area | Paperclip | Mission Control | Impact |
|------|-----------|-----------------|--------|
| Page entity name size | ~28-32px, commanding | 20px, blends in | **High** |
| Avatar/icon contrast | Visible, weighted | Disappears into background | Medium |
| Header layout balance | Left content + right action zone | Left-heavy, empty right | Medium |
| Content card hierarchy | Important sections visually distinct | All sections identical boxes | Medium |
| Sidebar section texture | Section types visually distinct, inline badges | Uniform flat list | Medium |
| Breadcrumb separator | `>` (directional) | `/` (path-like) | Low |
| Row trailing element density | Clean: ID + title + badge | Crowded: 4+ trailing elements | Low |

**Bottom line:** The biggest styling gap is typographic hierarchy. Paperclip uses size to tell you what's important — the agent name dominates the page. Our dashboard treats everything with the same visual weight, which makes it harder to scan. The fix is straightforward: make entity names bigger, make the most important status card visually distinct, and balance header layouts.
